# План оптимизации загрузки фото и видео

## Что уже сделано в проекте

- В `next.config.ts` включены современные форматы изображений (`avif`, `webp`) и задан `minimumCacheTTL` на 7 дней.
- В блоке `GetInTouch` видео подгружается только после попадания секции в зону видимости через `IntersectionObserver`.

## Узкие места сейчас

1. **Hero-видео на странице ремонта грузится сразу**: 
   - `src/app/components/services/apartment-renovation/RenovationHero.tsx` рендерит `<video>` сразу, без lazy-strategy.
2. **Есть `unoptimized` у изображений логотипов**:
   - В `src/app/components/home/complexes/index.tsx` много `Image ... unoptimized`, из-за чего обходится встроенная оптимизация Next.js.
3. **Нет единого стандарта размеров (`sizes`)**:
   - В части компонентов `Image` задана ширина/высота, но не везде есть точный `sizes`, что может приводить к скачиванию избыточных изображений на мобильных.
4. **Видео-источники не адаптивные по качеству**:
   - Используется одиночный MP4 (и локально, и с внешнего CDN), без вариаций битрейта/разрешения и без WebM fallback.

## Практический план (по приоритетам)

## P0 — быстрый выигрыш (1–2 дня)

1. **Отключить `unoptimized` там, где это безопасно**
   - Для локальных `public/images/**` перейти на стандартный `next/image` без `unoptimized`.
   - Для SVG-логотипов, если оптимизация не нужна, оставить `unoptimized` только точечно и обоснованно.

2. **Уточнить `sizes` для всех above-the-fold и карточек**
   - Для каждой сетки/карточки прописать реальный `sizes`, чтобы браузер не выбирал слишком крупный ресурс.
   - Пример: `(max-width: 768px) 92vw, (max-width: 1200px) 45vw, 30vw`.

3. **Видео: default `preload="none"` + `poster`**
   - Для фоновых видео добавить постер (`.webp`) и не грузить медиаданные заранее.
   - Для устройств с `prefers-reduced-motion` рендерить только статичное изображение.

## P1 — средний эффект (3–5 дней)

1. **Адаптивное видео (ABR-lite)**
   - Подготовить минимум 2 версии ролика: `720p` и `1080p`.
   - На клиенте выбирать источник по `matchMedia`/ширине экрана и `Save-Data`.

2. **Перевести внешнее видео на контролируемый CDN/хранилище**
   - Чтобы управлять cache headers, версионированием, и не зависеть от стороннего домена.

3. **Blur placeholders для ключевых фото**
   - Добавить `placeholder="blur"` и `blurDataURL` для крупных hero/карточек.

## P2 — системная оптимизация (1–2 недели)

1. **CI-проверка веса медиа**
   - Скрипт, который падает при добавлении изображения выше, например, 500KB (или video > 5MB для hero-loop).

2. **Перевод части контента в AVIF/WebP-цепочку на этапе сборки**
   - Генерация нескольких размеров заранее для самых популярных блоков.

3. **Наблюдаемость Web Vitals**
   - Отслеживать LCP/INP/CLS и коррелировать с медиа-блоками.

## Минимальные технические стандарты для команды

- Фото:
  - `next/image` по умолчанию.
  - Обязательные `width/height` (или `fill` + контейнер с фиксированным соотношением).
  - Обязательный `sizes`.
- Видео:
  - `preload="none"` для декоративного фона.
  - Обязательный `poster`.
  - Lazy mount через `IntersectionObserver` для блоков ниже первого экрана.
- Контроль веса:
  - Hero image: до ~200KB в мобильной версии.
  - Card image: до ~120KB.
  - Background loop video: желательно до 2–4MB для стартовой версии.
